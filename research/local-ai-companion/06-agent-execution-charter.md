# Agent 執行規範書 — 在 Mac mini 或筆電上跑 LiveTalking POC

---

## 0. TL;DR（給人類決策者，動手前先看這段）

在往下讀完整規範之前，先看這個事實，因為它會決定你該準備哪一台機器：

> **Mac mini（不論 Intel 版還是 Apple Silicon／M 系列版）在硬體上就是無法跑 CUDA。** 這不是「機率
> 偏低」，是確定的事實：Apple 從 2021 年起的 macOS 已完全不支援 NVIDIA 顯卡與 CUDA driver，而
> Apple Silicon 的 GPU（Metal/MPS 架構）跟 NVIDIA CUDA 是兩套完全不同的底層技術。而本次選定的
> `lipku/LiveTalking` 官方文件白紙黑字寫明是以 **CUDA 12.8 + NVIDIA GPU** 為前提測試（Ubuntu 22.04）。
> 官方 repo 的相容性徽章雖然寫著支援 `mac`，但**通篇 README／FAQ 找不到任何 macOS 安裝步驟、任何
> Mac 上的實測效能數據**——這是「宣稱支援但沒有驗證證據」的狀態，不能當作可靠依據。

**建議**：如果你的目的是**盡快跑出真人數位人即時直播的 POC**，優先準備**一台有 NVIDIA 獨立顯卡的
Windows 或 Linux 筆電**（RTX 3060 以上），這是本計畫可行性評估 ~91% 的那條路。Mac mini 若你已經有、
不想額外花錢買 PC，可以當作**次要、低優先的可行性測試**（本規範書的 Track B），但要有心理準備：
**大概率無法達到即時（≥25fps）效果，甚至可能連跑都跑不起來**，不要把它當作主要賭注。

以下規範書同時涵蓋兩條路，讓 Agent 自己偵測硬體後分流執行，你不需要現在就做決定。

## 1. 任務目標（Mission —— 唯一、明確）

你（執行任務的 agent）的任務是：

> 在**這一台**被指派的機器上，把開源專案 [`lipku/LiveTalking`](https://github.com/lipku/LiveTalking)
> 的 demo 跑起來，驗證「使用者打字輸入一段文案 → 系統用語音講出來 → 畫面呈現真人樣貌的數位人、嘴型
> 同步」這件事在**這台機器**上做不做得到，並且如實回報結果。

**目標的驗收定義（四個條件，缺一都算「未完全達成」，但要如實回報做到哪裡，不要美化）**：

| # | 條件 |
|---|---|
| A | 全程本地執行，不依賴雲端 API（可用斷網測試驗證） |
| B | 使用者打字輸入文案，系統用語音講出來 |
| C | 講話的是真人樣貌的數位人，嘴型與語音同步 |
| D | 畫面可以被推到直播管道（虛擬攝影機/RTMP 皆可） |

**你的任務是「驗證＋如實回報」，不是「不計代價讓它成功」。** 如果這台機器硬體不支援，正確的任務
完成方式是**清楚地回報「在這台機器上不可行，原因是 X」**，而不是想辦法繞過硬體限制硬做（見第 4 節
禁止事項）。這一點請認真對待——一份誠實的「不可行」報告，比一份灌水的「假裝成功」報告有價值得多。

## 2. Gate 0 — 硬體偵測與分流（第一步，強制執行，不可跳過）

在做任何安裝動作之前，**先偵測這台機器的硬體**，依結果分流到 Track A / B / C。這一步決定你後面
該花多少時間、預期什麼結果——跳過這步、直接開始安裝，是本規範書明確禁止的行為。

### 2.1 偵測指令

```bash
# 通用：作業系統與架構
uname -a

# macOS 專用（若 uname 顯示 Darwin）
sw_vers                                    # macOS 版本
sysctl -n machdep.cpu.brand_string         # CPU 型號（判斷 Intel Mac 還是 Apple Silicon）
system_profiler SPDisplaysDataType         # 顯示卡資訊（確認是否有獨立顯卡——Mac mini 基本上沒有）

# Windows / Linux 專用：檢查 NVIDIA GPU
nvidia-smi
```

### 2.2 分流判斷表

| 偵測結果 | 分流 | 預期結果誠實聲明 |
|---|---|---|
| `uname` 顯示 `Darwin`（macOS，包含 Mac mini） | **Track B** | 官方文件對 macOS **無任何驗證過的安裝步驟或效能數據**。CUDA 在 macOS 上**確定不可行**（Apple 平台限制，非本機硬體問題）。即使 LiveTalking 聲稱支援 Mac，也應假設「未經驗證」，用低期望值執行 |
| `nvidia-smi` 有正常輸出，顯示 NVIDIA GPU 型號與 VRAM | **Track A** | 這是官方驗證過的路徑，直接照 `05-poc-execution-plan.md` 執行，可行性 ~91% |
| Windows/Linux 但 `nvidia-smi` 找不到指令或無輸出（只有內顯或 AMD 顯卡） | **Track C** | 與 Track B 相同的「確定不可行」結論——CUDA 12.8 是官方硬性前提，沒有 NVIDIA GPU 就沒有這條路 |

**Track A 是本計畫唯一「預期會成功」的路徑。Track B/C 是「基於使用者現有設備，退而求其次去驗證看看」
的探索性任務，允許失敗，且失敗是正常且可預期的結果。**

## 3. 分流後的任務清單

### 3.1 Track A（偵測到 NVIDIA GPU）

直接執行 `research/local-ai-companion/05-poc-execution-plan.md` 的 Stage 0 → 1 → 2 → 3，逐步照做，
**不要重新設計步驟**，那份文件已經是逐指令的規格。完成後把結果填進該文件第 9 節的執行記錄表。

**本規範書對 Track A 的唯一補充**：即使是「預期會成功」的路徑，仍要遵守本文件第 4、5、6 節的行為
邊界與時間預算，不能因為信心高就跳過安全規則。

### 3.2 Track B / C（沒有 NVIDIA GPU，含所有 Mac）

這是**降規格的可行性探索**，目標不是「跑到即時直播」，而是「搞清楚這台機器到底能撐到哪一步」。
**總時間預算上限 2 小時**（見第 5 節），不要超支硬啃。

1. **Clone 專案並嘗試安裝**（依 `README.md` 的一般安裝步驟走一次，Python 3.12 + 建虛擬環境 + `pip
   install -r requirements.txt`）。記錄：哪些套件裝不上、什麼錯誤訊息。
2. **嘗試啟動服務**，觀察是否有 CUDA 相關的硬性 import 錯誤（例如 `torch.cuda` 相關的
   `AssertionError` / `RuntimeError`，或缺少 CUDA 相關動態函式庫）。這種錯誤代表**這條路徑對這個
   模型/程式碼路徑而言，在沒有 CUDA 的機器上是結構性不可行**，不是「裝錯東西」的問題。
3. **不要**為了讓它動起來去做以下事情（時間效益太低，且超出 POC 範圍，見第 4 節）：
   - 改寫原始碼把 CUDA-only 運算子換成 CPU/MPS 相容版本（這是深度移植工程，可能要花好幾天，且
     即使做完也達不到即時效能，不符本次 POC 的時間預算）
   - 反覆嘗試各種第三方 fork／未經驗證的修改版本
   - 安裝來源不明的替代套件企圖繞過官方相依需求
4. **如果**環境裝得起來、服務也真的啟動了（不代表達到即時，只代表「有跑」），才進一步照
   `05-poc-execution-plan.md` Stage 2 的 `curl` 指令測一次文字→語音；若跑得動但明顯很慢（例如
   一句話要等 10 秒以上才有反應），這本身就是一個有效的資料點，記下來即可，**不需要**再去調校
   效能。
5. 產出第 7 節格式的報告，明確寫出「這台機器能撐到清單裡的哪一步、卡在哪、為什麼」。

## 4. 明確禁止事項（不會做的事，優先序高於任何「把它做成功」的衝動）

無論 Track A 或 B/C，以下事項**一律不可執行**，遇到需要才能繼續時，停下來回報並等待人類指示，
**不要自行判斷「應該沒關係」就繼續**：

1. **不公開直播**。畫面只能連到本機瀏覽器（`localhost`）或本機虛擬攝影機做驗證。不得推流到任何
   真實的 YouTube/Twitch/TikTok 等平台帳號，即使是「私人」/「未公開」模式也需要先問過人類——直播
   平台帳號是否使用、用哪個帳號，是人的決定，不是 agent 的決定。
2. **不使用未經授權的真人素材**。不得用任何非使用者本人、且未取得明確授權的真實人物照片/影片/聲音
   當作形象或聲音克隆來源。本次 POC 一律使用專案內建的 demo avatar/demo 聲音，不去網路上找真人素材。
3. **不上傳個資到第三方雲端服務**。整個流程應保持本地執行；不得把使用者的照片、聲音樣本、對話內容
   上傳到任何雲端 API（除非該工具的官方預設行為本就如此且用途僅限初次安裝時的套件/模型下載，例如
   從 GitHub/Google Drive 下載公開模型權重——這類「取得公開素材」不算違反本條，「上傳使用者個資」
   才算）。
4. **不花錢**。不得訂閱任何付費 API、不得租用雲端 GPU 主機、不得在任何地方輸入信用卡資訊，除非事先
   明確告知人類並取得同意。
5. **不做風險性的系統層級變更**：不關閉防火牆、不對外開放非必要 port、不刪除使用者機器上與本任務
   無關的檔案或其他專案的環境。需要 `sudo` 權限的安裝動作，先列出確切指令，讓人類確認後再執行（若
   agent 本身就是在無人值守模式下運作，至少要把每一條 `sudo` 指令完整記錄進報告）。
6. **不做超出 POC 範圍的原始碼移植工程**（見 3.2 第 3 點）。
7. **不自動把結果 commit/push 回這個 `Buildup_KOL` git repo 的任何分支**，包括 `main`。只把報告寫成
   本地檔案（見第 7 節），是否要同步回 repo、開 PR，由人類決定並另外執行。
8. **不假裝驗證了自己實際上驗證不了的事**（見第 6 節「人類需親自確認 vs 機器可自行確認」）。

## 6. 「機器可自行確認」vs「需要人類確認」——回報時務必分開寫

很多 coding agent（包括負責跑這份規範的你）**沒有能力真正「聽到聲音」或「看到影片裡嘴型準不準」**。
回報結果時，把驗收判準分成兩類，不要把兩類混在一起說成「已驗證通過」：

**機器可自行確認**（你可以直接看指令結果判斷）：
- 服務啟動沒有 exception / crash
- HTTP API 呼叫回傳 `200` / `{"code": 0}` 等預期回應
- log 裡有印出 `inferfps`/`finalfps` 數值，可以讀出數字判斷是否 ≥25
- 進程持續運行、沒有中途掛掉
- 斷網後 API 呼叫是否仍成功（機器可判斷成功/失敗，但**判斷不了聲音是不是真的用本地引擎產生**，
  只能判斷「有沒有回應」）

**需要人類親自確認**（你必須明確請人類去看/聽，不能替他打勾）：
- 「聽起來像不像真人講話」「嘴型跟聲音準不準」這類主觀品質判斷
- 直播畫面在 OBS/平台端實際呈現的樣子
- 聲音克隆像不像本人（若做到 Stage 4）

回報時請用這樣的句型，不要含糊帶過：「機器面：服務正常啟動、`/human` API 回應 200、`finalfps` 記錄
為 27（≥25 達標）。**需要你親自確認**：請打開 `http://localhost:8010/index.html` 聽一下語音、看一下
嘴型是否同步，這部分我沒有能力代替你判斷。」

## 7. 完成定義與回報格式

不論 Track A 或 B/C、不論成功或失敗，**任務結束時都必須產出一份結構化報告**（可直接沿用/擴充
`05-poc-execution-plan.md` 第 9 節的表格），至少包含：

```
執行日期：__________
Gate 0 偵測結果：Track A / B / C（圈選），硬體資訊：__________
本次時間花費：__________（是否超過第 5 節預算：是/否）

機器可自行確認的結果：
  - 環境建置：通過 / 卡在哪一步 __________
  - 服務啟動：通過 / 失敗，錯誤訊息 __________
  - API 回應：__________
  - inferfps / finalfps：__________（≥25：是/否）
  - 斷網測試（條件 A）：__________

需要人類確認的項目（尚未確認，請你親自檢查）：
  - [ ] 語音是否聽得到、聽起來自然
  - [ ] 嘴型是否與語音同步
  - [ ] （若做到 Stage 3）直播輸出畫面是否正常

整體結論：四條件 A/B/C/D 在這台機器上，哪些已驗證可行、哪些不可行、原因：
  __________

下一步建議（例如「這台機器不可行，建議改用 XXX」或「已可行，建議進到 04 文件 Stage 4」）：
  __________
```

這份報告本身就是任務的最終產出，寫成本地 markdown 檔案（例如
`research/local-ai-companion/results/{日期}-{機器代號}-poc-report.md`），**不要自動 commit/push**
（見第 4 節第 7 點），交回給人類決定後續。
