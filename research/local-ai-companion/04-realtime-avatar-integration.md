# Phase 2/3 決策文件 — 真人數位人「文字→語音→直播」整合藍圖

> **Status（2026-07-10）**：本文件回答一個明確目標——「找出一個**本地可立即整合、可行性最高**的
> 現成產品/開源專案」，做到：**本地輸入文字 → 用（真人的）聲音講出來 → 由真人樣貌的數位人呈現 →
> 之後能直播出去**。經過候選盤點與可行性評估，**選定 [`lipku/LiveTalking`](https://github.com/lipku/LiveTalking)
> 作為主整合對象**，並以它為核心把架構建立起來。
>
> **與先前文件的關係／一個重要轉向**：`01`/`02`/`03` 為「本地離線」考量，預設偏向 **Live2D/VRM
> 風格化**虛擬形象。本文件依使用者明確要求「**它是真人的**」，把 Phase 2/3 的形象路線**改為真人
> talking-head（音訊驅動對嘴）**。此轉向僅影響「形象呈現層」；`03` 的 persona → prompt、Orchestrator
> 對話核心不受影響，反而成為本方案的「chat 模式」上游（見第 5 節）。

---

## 0. 目標與驗收定義

使用者要的**單一結果**，拆成可驗收的四個條件：

| # | 條件 | 驗收方式 |
|---|---|---|
| A | 全程可在**本地**執行（不強制依賴雲端 API） | 斷網後仍能跑完 A→D |
| B | 我**打字輸入**一段文案，系統把它**講出來**（語音） | 輸入文字，喇叭/串流聽到對應語音 |
| C | 講話的是**真人樣貌**的數位人（不是動漫/2D 卡通） | 畫面是真人臉、嘴型與語音同步 |
| D | 這個畫面**能被直播出去** | 能推到 OBS / RTMP / 虛擬攝影機，平台端看得到 |

**可行性門檻**：整體可行性需 > 80%。本文件第 2 節給出評分。

---

## 1. 候選盤點（為什麼是 LiveTalking）

盤點「本地 + 真人 talking-head + 可即時輸出」這個交集下的現成方案：

| 方案 | 本地離線 | 真人臉 | 文字→語音 | **即時直播輸出** | 定位 | 對本目標適配 |
|---|---|---|---|---|---|---|
| **LiveTalking（lipku）** | ✅ | ✅ Wav2Lip/MuseTalk | ✅ 內建 TTS + echo/chat | ✅ **原生 RTMP/虛擬攝影機/WebRTC** | 即時互動數位人**直播引擎** | **★ 最佳** |
| HeyGem（Duix/硅基） | ✅ 全離線桌面 App | ✅ 影片克隆 | ✅ 文字/語音驅動 | ❌ 以**離線影片生成**為主，非即時串流 | 離線影片產生器 | 次選（做預錄片，不做即時直播） |
| Open-LLM-VTuber | ✅ | ❌ Live2D 風格化 | ✅ | ⚠️ 桌寵為主 | 語音陪伴 VTuber | 不符「真人」 |
| SadTalker / Hallo 等 | ✅ | ✅ | 需另接 | ❌ 批次生成、非即時 | 研究型 talking-head | 不符「即時直播」 |
| 商用 API（HeyGen/Beyond Presence/D-ID） | ❌ 雲端 | ✅ | ✅ | ✅ | SaaS | 不符「本地」硬需求 |

**結論**：只有 **LiveTalking** 同時滿足 A、B、C、D 四項且是**單一 repo 打通全鏈**，其餘方案都在某一項
硬需求上出局或需要大量自行拼裝。HeyGem 留作「需要高品質預錄影片」時的搭配工具（見第 7 節備援）。

> 佐證（來自 Evelyn 報告 `references/AI_Livestream_Report.md`）：商用直播主播端到端語音回應約 1 秒、
> 渲染 <200ms，且**動作生成不成熟、主流靠預錄片庫觸發**——這與 LiveTalking 的「即時對嘴 + 動作編排
> （action orchestration）觸發預錄動作」設計一致，代表 LiveTalking 走的是業界已驗證的路線。

## 2. 可行性評估（> 80% 門檻）

### 2.1 逐條證據（全部取自 LiveTalking 官方 README）

- **模型（真人臉，音訊驅動）**：Wav2Lip、MuseTalk、ER-NeRF、Ultralight-Digital-Human。
- **實測效能（消費級 GPU 即可即時）**：
  - Wav2Lip256 @ **RTX 3060 = 60 FPS**、@ RTX 3080Ti = 120 FPS
  - MuseTalk @ RTX 3090 = 45 FPS、@ RTX 4090 = 72 FPS
  - 即時定義：GPU 推理 FPS 與最終推流 FPS **皆 ≥ 25** 才算即時（`inferfps`/`finalfps`）。
- **文字輸入**：`/human` API，`echo`（直接復讀你打的文案）與 `chat`（接 LLM 對話）兩種模式。
  → **echo 模式 = 條件 B 的完美命中**：打字進去，講出來。
- **語音（可真人聲音）**：模組化 TTS，支援 EdgeTTS（開箱即用）、**GPT-SoVITS / CosyVoice（聲音克隆）**、
  騰訊雲等；且平台層級「支援聲音克隆」。
- **直播輸出**：**WebRTC / RTMP / 虛擬攝影機**三選一。→ 條件 D 命中（RTMP 直推平台，或虛擬攝影機餵 OBS）。
- **真人形象來源**：`/avatar.html` 上傳一段影片即自動生成數位人形象。→ 條件 C 命中。
- **其他**：可打斷、全身影片拼接、動作編排、多路併發、自訂形象。
- **技術棧**：Python 3.12 / PyTorch 2.9.1 / CUDA 12.8（Ubuntu 22.04 實測）。
- **社群**：GitHub ~4.3k★、持續維護、文件完整。

### 2.2 評分

| 面向 | 權重 | 評分 | 說明 |
|---|---|---|---|
| 功能覆蓋 A/B/C/D | 40% | 100% | 四條件單一 repo 全覆蓋，無缺口 |
| 即時性可達 | 20% | 95% | 消費級 GPU 已有官方 FPS 實測達標 |
| 整合難度（拿來即用程度） | 20% | 80% | 有 API/前端頁，但 CUDA/PyTorch 環境建置有摩擦（社群回報「dependency hell」） |
| 授權/可商用 | 10% | 75% | repo 本體開源；但真人**肖像與聲音來源**需自備且合規（見第 8 節） |
| 硬體門檻 | 10% | 85% | 需 NVIDIA GPU；Wav2Lip 路線 RTX 3060 即可，門檻不高 |

**加權可行性 ≈ 91%**（前提：手上有一張 **NVIDIA GPU（≥ RTX 3060）**）。

> **把 91% 推向 100% 的唯一關鍵輸入 = 你的 GPU 型號**。
> - 若是 **RTX 3060–3070（8–12GB）**：走 **Wav2Lip256** 路線，60fps 綽綽有餘，可行性視為 ~95%。
> - 若是 **RTX 3080Ti / 3090 / 4090 及以上**：可走 **MuseTalk**，畫質更好、仍即時，可行性 ~95%。
> - 若**沒有 NVIDIA GPU（純 CPU / Mac / AMD）**：即時真人 talking-head 不可行，需退到 HeyGem 離線
>   出片（非即時）或改用雲端——這是唯一會讓可行性掉到門檻以下的情境。
>
> 本文件的架構**同時設計了 Wav2Lip 與 MuseTalk 兩條路徑**，因此在確認 GPU 前即可先動工，不阻塞。

**結論：可行性 > 80% 達成（估 ~91%，有 NVIDIA GPU 前提下）。以 LiveTalking 為核心建立架構如下。**

## 3. 目標架構（以 LiveTalking 為核心）

```
┌──────────────────────────────────────────────────────────────────────────┐
│  你（本地操作者）                                                          │
│  ①打字輸入文案 ──────────────────────────────────────────────┐            │
└───────────────────────────────────────────────────────────────┼────────────┘
                                                                 │ text
        （可選上游）                                              ▼
  ┌───────────────────────────┐   chat 模式文字流   ┌────────────────────────────┐
  │ companion Orchestrator     │ ─────────────────▶ │  LiveTalking Server         │
  │ （03 文件的對話核心，      │                     │  ─ /human API（echo|chat）  │
  │   接 kols/{id} persona）   │   ◀── echo 模式 ──  │  ─ TTS：EdgeTTS/GPT-SoVITS  │
  │  ※ MVP 可先跳過，直接 echo │   直接送文案文字     │       /CosyVoice（聲音克隆）│
  └───────────────────────────┘                     │  ─ 對嘴：Wav2Lip / MuseTalk │
                                                     │       （真人參考影片）      │
   kols/{id} 的真人臉素材 ─────────────────────────▶ │  ─ 形象：/avatar.html 上傳  │
   （影片，作 avatar 來源）                           └──────────────┬─────────────┘
                                                                    │ 音視訊（真人臉+語音，已對嘴）
                                              ┌─────────────────────┼─────────────────────┐
                                              │ ②RTMP 直推           │ ②虛擬攝影機          │ ②WebRTC
                                              ▼                      ▼                     ▼
                                    直播平台（YT/Twitch/         OBS →（疊字幕/場景）        瀏覽器即時預覽
                                    TikTok/Shopee Live…）        → 再直播到平台
```

### 3.1 三層職責

1. **輸入層**：你打字。MVP 走 **echo 模式**——文字直接進 `/human`，LiveTalking 用 TTS 唸出來、驅動
   真人臉。這就是條件 A+B+C 的最短路徑。
2. **LiveTalking 引擎層**（核心，不自建）：TTS（文字→語音，可用聲音克隆做「你/角色的真人聲音」）+
   對嘴模型（語音→真人臉嘴型）+ 串流輸出。
3. **直播輸出層**：三種出口，建議**虛擬攝影機 → OBS → 平台**（最有彈性：可加字幕、場景、商品卡），
   或需要極簡時直接 **RTMP** 推平台。

### 3.2 為什麼這個切法能同時滿足「MVP 快」與「未來完整」

- **MVP（echo）**：不需要 LLM、不需要我們的 Orchestrator，打字即說即播，最快驗證 A/B/C/D 全鏈打通。
- **完整（chat）**：把 `03` 文件的 Orchestrator（讀 `kols/{id}` persona、串流輸出文字）接到
  LiveTalking 的 `chat` 模式或改用 echo 餵它——就從「照稿唸」升級成「有人格、會即時對話的真人數位人
  直播」。兩種模式共用同一套引擎與直播出口，不重做。

## 4. 硬體與部署路徑（依 GPU 分兩條，先動工不卡）

| GPU 情境 | 對嘴模型 | 預期 FPS | TTS 建議 | 備註 |
|---|---|---|---|---|
| RTX 3060 / 3070（8–12GB） | **Wav2Lip256** | ~60fps | EdgeTTS 起步 → GPT-SoVITS 克隆 | 入門即達即時，CP 值最高 |
| RTX 3080Ti / 3090 / 4090（≥12–24GB） | **MuseTalk** | 45–72fps | GPT-SoVITS / CosyVoice | 畫質/口型更佳 |
| 無 NVIDIA GPU | —（即時不可行） | — | — | 退 HeyGem 離線出片或雲端，見第 7 節 |

- **作業系統**：官方在 Ubuntu 22.04 實測；Windows 可行但依賴建置更麻煩，建議 Linux 或 WSL2。
- **環境**：Python 3.12 + PyTorch 2.9.1 + CUDA 12.8。建議用 conda/venv 隔離，避免污染系統環境。
- **與本地 LLM 的 VRAM 競合**：若同機又要跑 `03` 文件的本地 LLM（chat 模式），兩者搶同一張卡的
  VRAM。MVP echo 模式不吃 LLM，衝突最小；chat 模式建議：對嘴用 Wav2Lip（省 VRAM）+ 小一點的 LLM，
  或把 LLM 放另一台/另一張卡。

## 5. 與現有資產（kols/ + 03 設計）的整合點

1. **真人形象來源**：本 repo 的 KOL 目前是 **AI 生成的靜態圖/影片**（higgsfield 等）。要當 LiveTalking
   的 avatar，需要該 KOL 的**一段真人樣貌影片**（可由既有 AI 影片產線先產一段 talking 素材，再上傳
   `/avatar.html`）。這把「靜態 KOL IP」延伸成「會說話的真人數位人」，與 repo 主線一致。
2. **聲音**：用 GPT-SoVITS/CosyVoice 對「該角色的參考語音」做**聲音克隆**，讓語音與角色人設一致
   （對應 `kols/{id}` 的 `voice_tone` 設定）。
3. **人格（chat 模式）**：`03` 文件的 Orchestrator 輸出（讀 `kols/{id}/profile.json` + `character.md`
   組出的人格化文字，token streaming）→ 餵 LiveTalking。`03` 早已要求 streaming 輸出，正好銜接即時
   語音。**echo 模式 MVP 階段可先不接這層。**
4. **不衝突宣告**：本方案不改 `03` 的 Orchestrator/persona 設計，只是在它「下游」接一個真人形象+語音
   的呈現與直播出口；`03` 的 CLI 也仍可獨立作為純文字驗證工具。

## 6. 落地步驟（可直接執行的驗證計畫）

> 目標：以最短路徑證明「打字 → 真人臉開口說 → 能直播」端到端成立（條件 A–D）。

1. **環境**：Ubuntu 22.04（或 WSL2）+ conda 建 Python 3.12 環境，裝 CUDA 12.8 / PyTorch 2.9.1。
2. **取得 LiveTalking**：`git clone https://github.com/lipku/LiveTalking`，依 README 裝依賴、下載
   Wav2Lip（或 MuseTalk）權重。
3. **最小 echo 驗證**：用內建示範 avatar + EdgeTTS 啟動 server，呼叫 `/human`（echo 模式）送一句
   中文文案，確認**瀏覽器 WebRTC 預覽**看到真人臉唸出這句話（打通 A+B+C）。
4. **直播出口驗證**：改用**虛擬攝影機**輸出，在 OBS 選該虛擬攝影機為來源，OBS 推流到測試用 RTMP／
   平台，確認觀眾端看得到（打通 D）。
5. **換成「我們的角色」**：`/avatar.html` 上傳選定 KOL 的真人樣貌影片生成形象；用 GPT-SoVITS 克隆該
   角色聲音替換 EdgeTTS。此時輸出即為「該 KOL 的真人數位人照稿直播」。
6. **（可選）升級 chat**：把 `03` Orchestrator 接上 `/human` chat 模式，變成可即時互動的人格化直播。
7. **驗證產出**：錄一段 echo 端到端 demo，記錄實測 `inferfps`/`finalfps`（需 ≥25）與端到端延遲，回填
   本文件「實測」欄。

**第一個試跑角色建議**：選 `kols/` 中資料最完整、且形象素材最容易產真人影片者（如 `chloe-lin` 或
`sienna-lai`）——此決策與 `03` 文件的「試跑角色待決策」共用結論即可。

## 7. 備援方案（GPU 不足 / 需要高品質預錄時）

- **HeyGem（Duix/硅基，全離線）**：不吃即時 GPU 串流壓力，適合**預先生成**高品質真人講稿影片
  （條件 A+B+C，但**非即時**、不直接直播）。工作流：HeyGem 出片 → OBS 播放清單/媒體源 → 直播。
  適合「不需要即時互動、只要真人數位人照稿講」的直播型態，是 LiveTalking 之外的穩健 fallback。
- **雲端（Beyond Presence / HeyGen 等）**：僅在完全沒有本地 GPU 且可接受非本地時考慮，違反條件 A，
  列此僅為完整性。

## 8. 風險與注意事項

1. **GPU 是硬前提**：無 NVIDIA GPU 則即時路線不成立（見第 2.2、第 4 節退路）。**唯一需要你確認的輸入。**
2. **依賴建置摩擦**：CUDA/PyTorch 版本敏感，社群回報安裝踩坑多；用官方指定版本 + 隔離環境可大幅降低。
3. **肖像與聲音合規（重要）**：真人形象與聲音克隆涉及**肖像權/聲音權/知情同意**。用於本 repo 的 AI KOL
   時，形象/聲音應來自「可授權使用的素材或本專案自產的 AI 角色影像」，不得未經同意克隆真實他人。
4. **AI 揭露合規**：多個市場要求直播主播為 AI 生成時須揭露（見 Evelyn 報告第 5 節）。直播上線前確認
   目標平台/地區規範。
5. **動作/舞蹈**：即時原創動作生成不成熟，如需肢體動作用「預錄動作片庫 + 動作編排觸發」（LiveTalking
   支援 action orchestration），勿期待文字直接生成任意舞蹈。
6. **VRAM 競合**：chat 模式同機跑 LLM + 對嘴要分配好 VRAM（見第 4 節）。

## 9. 決策摘要（給後續 agent 與使用者）

- **選定整合對象**：`lipku/LiveTalking`（可行性 ~91%，> 80% 門檻達成）。
- **MVP 路徑**：echo 模式 + Wav2Lip + EdgeTTS + 虛擬攝影機→OBS，最短證明 A–D。
- **完整路徑**：GPT-SoVITS 聲音克隆 + KOL 真人形象 + `03` Orchestrator chat 模式。
- **唯一待你確認**：**你的 GPU 型號**（決定 Wav2Lip vs MuseTalk，並把可行性推向 ~95%+）。
- **後續執行**：由 `livestream-tech-specialist` 依第 6 節落地步驟展開，並回填實測數據。
