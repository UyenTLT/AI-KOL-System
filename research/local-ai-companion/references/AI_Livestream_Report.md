## 這份報告在講什麼

聚焦「AI 直播帶貨」場景：用合成數位主播即時廣播、賣貨、與觀眾互動。它把系統拆成兩根技術支柱，
再補上商用平台調查、電商實戰案例、以及一份原型技術棧建議：

1. **即時可控 AI 虛擬形象生成**（會說話、比手勢、甚至跳舞）
2. **觀眾留言辨識 → 映射成虛擬形象行為**的互動迴圈

## 對本專案的價值定位

這份報告的場景（**電商直播帶貨、雲端 API、多 GPU**）與我們 Phase 1 的目標（**單機、本地、純文字、
一對一 AI 伴侶**）方向不同——它偏「雲端商用、對眾廣播」，我們偏「本地私有、對單一使用者互動」。
**因此它不改變 Phase 1 的任何決策**，但它是 Phase 3（把 AI 伴侶做成可被看見、甚至公開直播的
虛擬 KOL）非常好的產業現況地圖與待踩雷區清單。把它當成 Phase 3 的「別人已經走過的路」，而不是
Phase 1 的施工圖。

## 重點整理（含對應到我們框架的註記）

### 支柱一：即時虛擬形象與影片生成
- **Diffusion 即時串流虛擬形象**：代表性系統用 14B diffusion 模型、5 張 H800、4-step sampling
  做到 45fps 的無限長度串流輸出。→ 註：這是**雲端多 GPU 規格**，與我們單機本地前提相衝，僅供了解
  技術上限；Phase 3 若走本地，仍應優先 Live2D/VRM（見我們 01 文件第 4 節），而非 diffusion 真人臉。
- **商用虛擬形象平台**：Beyond Presence（1080p、<100ms 延遲、frame-accurate 對嘴、可外接
  OpenAI/ElevenLabs/Cartesia/Hume 語音）、LiveAvatar（最少 2 分鐘素材即可建 avatar，可整包或只租
  串流層）、HeyGen/Synthesia/D-ID/DeepBrain（更全面的 AI 影片/虛擬形象套件，部分有對話互動模式）。
  → 註：LiveAvatar / HeyGen 我們 01 文件第 4 節已提及，這份報告補上了 Beyond Presence 與更完整的
  商用清單，可回填我們的調查。
- **MSI DigiME**：純 webcam 的 2D→3D 體感追蹤驅動 3D VRM 虛擬形象，免動捕硬體，且原生整合
  Twitch/YouTube 直播平台。→ 註：這對 Phase 3「低成本本地驅動 VRM」是很務實的一條路，值得
  `livestream-tech-specialist` 後續深入。
- **語音合成**：ElevenLabs / Cartesia / Hume，強調**低端到端延遲是整條 pipeline 反應速度的閘門**。
  → 註：與我們一貫強調的「latency budget」一致；但這些是雲端 TTS，本地取向仍看我們 01 文件的
  Coqui-TTS / GPT-SoVITS。
- **動作/舞蹈生成**：報告明講這是**整個技術棧最不成熟的一環**——目前沒有成熟商用 API 能純靠文字/
  留言指令生成高品質原創舞蹈；主流做法是**觸發預錄動作片庫**。→ 註：Phase 3 若要「跳舞/比手勢」，
  務實做法是預錄動作庫 + 指令觸發，不要期待即時生成。

### 支柱二：留言辨識與互動回應 pipeline（五階段）
1. **留言擷取**：從 TikTok/抖音/Facebook/YouTube/Shopee Live 等平台的 chat/live-comment API 拉留言。
2. **語音/文字辨識**：語音走 ASR（廠商宣稱吵雜環境 >95% 準確率），文字留言跳過此步。
3. **意圖分類（NLU）**：用 LLM（GPT/Claude/Gemini/文心/豆包…）判斷每則留言意圖（問商品/問價/
   要求跳舞/閒聊/spam），並維持多輪上下文。
4. **行為路由（Action Routing）**：**通常客製自建、非現成**，把意圖映射到具體動作（觸發預錄舞蹈、
   拉商品資料回答價格、或用 LLM 生成對話回覆）。
5. **渲染與輸出**：音訊過對嘴 + 虛擬形象渲染推到直播流。廠商數據：渲染延遲 <200ms、端到端語音
   回應約 1 秒。
   → 註：這個五階段與我們 02/03 的分層**高度可對應**——第 3 步的 LLM 意圖分類其實就是我們
   Orchestrator 的延伸，第 4 步 Action Routing 是我們目前架構還沒有、但 Phase 3 一定要新增的一層
   （「留言 → 行為」的映射），值得記進 Phase 3 待辦。

### 案例（證明已達商用規模）
- **百度／羅永浩數位分身**（2025-06-15）：基於文心 + Xiling 引擎，2.5 小時直播、展示 30+ 商品、
  1300 萬+ 觀看、約 5,500 萬人民幣銷售，逼真到許多觀眾沒發現主播是合成的。
- **京東 618**（2025）：50+ 數位人直播、部分長時間連續運行，AI 主播據報達到真人主播約 80% 的
  銷售表現。
  → 註：這兩個案例說明「AI 虛擬主播帶貨」已是可規模化的商業現實，對本專案是市場面的佐證。

### 建議原型技術棧（報告版）
| 層 | 用途 | 報告舉例工具 |
|---|---|---|
| 虛擬形象渲染 & 對嘴 | 即時 talking-head 影片 | Beyond Presence、LiveAvatar、HeyGen |
| 語音合成 (TTS) | 文字轉自然語音/克隆聲 | ElevenLabs、Cartesia |
| 對話/意圖模型 | 理解留言、生成回覆、分類意圖 | Claude、GPT、Gemini |
| 行為路由 | 意圖→虛擬形象動作 | 客製應用邏輯 |
| 動作片庫 | 預錄手勢/舞蹈 | 客製片庫 |
| 即時傳輸 | 低延遲音視訊送到平台 | LiveKit、ZEGOCLOUD |

→ 註：這是**雲端/商用**取向的棧；我們本地取向的等價對照見 `01-landscape-existing-solutions.md`
第 4 節。兩份可並排看：報告告訴你「不在意本地化、要最快上線」時業界怎麼配；01 文件告訴你「堅持
本地離線」時開源怎麼配。

### 挑戰與限制（報告點名，全部對我們 Phase 3 有效）
- **動作生成不成熟** → 靠預錄片庫。
- **延遲堆疊**：ASR + LLM + TTS + 渲染逐層累加，需全鏈路優化。
- **恐怖谷**：近寫實虛擬形象若微表情/時機不準反而讓觀眾不適 → 這也是我們偏好風格化 Live2D/VRM
  而非真人臉 diffusion 的另一個理由。
- **內容審核**：留言流進 LLM/虛擬形象前必須先過濾 spam/濫用 → 對應我們 02 文件的「內容邊界」
  待決策，且在直播對眾場景比一對一更嚴苛。
- **揭露與信任**：多個市場正走向「必須揭露主播為 AI 生成」的合規要求 → Phase 3 若公開直播需納入
  合規考量。
